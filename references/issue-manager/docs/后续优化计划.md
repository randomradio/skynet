## 一、Labels 标签能力建设

### 原计划

**现状**：Issue 上打了很多 labels（GitHub 标签），但目前并未在分析中有效使用。

**目标**：
- 把所有 labels 用起来，分析要基于 labels 进行
- 产品方案中 labels 包含**客户信息**（如 `customer/金盘`），需要支持：
  - 按 labels 筛选出标记了某客户的 Issue
  - 对某客户相关的 Issue 单独做分析

### 实际调整情况

| 项 | 状态 | 说明 |
|----|------|------|
| labels 存储 | ✅ 已实现 | GitHub labels 已完整保存在 `issues_snapshot.labels` 字段 |
| 使用文档 | ✅ 已实现 | [Labels标签使用说明](Labels标签使用说明.md) 含 SQL/Python 查询示例 |
| 分析中应用 | ✅ 已实现 | 可扩展分析含 `label_analysis`、`customer_analysis`；多维度报告含客户维度 |

## 二、关联关系完整性

### 原计划

**现状**：全量同步时因处理顺序问题，部分关联关系丢失。

**目标**：所有 Issue 入库后再统一提取关联，避免因先后顺序导致的关联丢失。

### 实际调整情况

| 项 | 状态 | 说明 |
|----|------|------|
| 流程调整 | ✅ 已实现 | 采用 **Phase 1 + Phase 2** 流程 |
| Phase 1 | ✅ 已实现 | 采集、AI 解析、存储 Issue 和评论（暂不处理关联） |
| Phase 2 | ✅ 已实现 | 所有 Issue 入库后，统一提取并批量保存关联关系 |
| 补全脚本 | ✅ 已实现 | `scripts/supplement_relations.py` 供旧流程跑完后补充缺失的关联 |

**补全脚本使用**（在项目根目录执行）：
```bash
python3 scripts/supplement_relations.py
python3 scripts/supplement_relations.py --repo-owner matrixorigin --repo-name matrixflow
python3 scripts/supplement_relations.py --clear-first  # 先清空再补全
```

## 四、两条分析线（2026-02-24 方案整合）

### 第 1 条线：项目推进分析（高频 - 每日/每周）

按客户、层级分析进度，AI 输出健康度与建议：

```
客户：金盘
  ├─ L1: Customer Project
  ├─ L2: Features
  ├─ L3: Tasks
  └─ L4: Bugs

AI 输出：健康度评分、关键发现、堵塞分析、建议行动、紧急事项
```

**实现**：`modules/ai_analysis`，`scripts/run_ai_analysis.py`

### 第 2 条线：横向关联分析（低频 - 每周/每月）

- 共性 Feature：被多个客户共用的功能
- 高 Bug Feature：关联 Bug 较多的功能
- AI 输出：模式总结、战略建议、资源分配建议

**实现**：同上，`scripts/run_ai_analysis.py`（可选 `--skip-cross` 跳过）

**使用**：
```bash
python3 scripts/run_ai_analysis.py --repo-owner matrixorigin --repo-name matrixflow
```

**输出**：`data/reports/ai_analysis/project_progress_YYYYMMDD.json`、`cross_patterns_YYYYMMDD.json`

## 五、可扩展分析框架（已实现）

### 架构

```
analysis_config.yaml → ExtensibleAnalysisEngine → Analyzers → Formatters (JSON/MD/HTML)
```

### 已启用的分析器

| 分析器 | 说明 |
|--------|------|
| basic_stats | 基础统计（state、issue_type、priority、status） |
| label_analysis | 标签分析（kind/、area/、customer/、severity/） |
| module_analysis | 功能模块分析（从 area/ 标签提取） |
| hierarchy_analysis | 层级分析、孤立 Issue 识别 |
| customer_analysis | 客户维度、共用 Feature |
| relation_analysis | 关联分析、阻塞链、被引用最多的 Issue |
| trend_analysis | 时间趋势（7d/30d/90d） |

### 使用方式

```bash
python3 auto_run.py --repo-owner matrixorigin --repo-name matrixone --extensible-report --email user@example.com
python3 scripts/run_extensible_analysis.py --repo-owner matrixorigin --repo-name matrixone
```

### 配置与扩展

- 修改 `config/analysis_config.yaml` 可调整分析维度和输出格式，无需改代码
- 新增分析器：继承 `BaseAnalyzer`，实现 `analyze(issues) -> Dict`，在配置中启用即可

## 十、自动写 Issue 与 AI 看板（已实现 - 独立模块，待合并主项目）

以下两项在 **`feature_issue_and_kanban/`** 下独立开发，测试通过后再合并主项目。设计稿与 PRD 见 Download「自动写issue和AI看板功能设计稿-260225.zip」及「PRD-自动写Issue与AI看板-260225.md」。

### 优化项 1：AI 驱动 Issue 创建（自动写 Issue）

| 项 | 说明 |
|----|------|
| **目标** | 用户通过聊天提供描述与账号 → 按类型带出正文模板 → 生成本地预览（GitHub 风格，含 Labels/Type/层级）→ 确认后调用 GitHub API 创建；支持批量创建并输出 Excel 链接表。 |
| **方案要点** | ① 纯预览脚本 `generate_preview_only.py`（不依赖 DB/AI，保证能出预览页）；② `create_issue_interactive.py` 支持单轮/交互/直接传 `--title` `--body` `--type`（未传 body 时按类型加载模板）；③ 各类型正文模板放在 `templates/*.md`（Doc Request、User Bug、MOI Feature 等）；④ 创建 Issue 通过 `issue_creator/github_issue_creator.py` 调 GitHub API；⑤ 聊天流程：在 Cursor 中说明需求 → AI 收集信息 → 执行预览脚本并打开浏览器 → 用户确认后执行创建脚本并返回链接。 |
| **脚本与模块** | `generate_preview_only.py`、`create_issue_interactive.py`、`batch_create_issues.py`；`issue_creator/`（knowledge_extractor、ai_issue_generator、github_issue_creator）。 |
| **状态** | ✅ 已在 feature_issue_and_kanban 实现；预览含层级/类型/Labels；支持 --demo 与真实创建。 |

### 优化项 2：项目看板（AI 看板 / 每日看板）

| 项 | 说明 |
|----|------|
| **目标** | 按项目标签（如 `project/问数深化`）聚合 Issue，生成每日看板：总体进度、逾期项、阻塞项、进行中；输出与「项目计划-甘特图 v3」同风格的 HTML（顶栏渐变、任务表、状态徽章、风险评估卡片）；可选 AI 总结。 |
| **方案要点** | ① 数据来源：优先 `project_issues`，表不存在或无数据时回退从 `issues_snapshot` 按标签查询；② `generate_daily_dashboard.py` 支持 `--demo` 假数据、`--output-html` 输出甘特图风格 HTML；③ `DashboardGenerator.generate_dashboard_html()` 生成与甘特图 v3 一致的顶栏/表格/风险评估区；④ 同步脚本 `sync_project_issues.py` 将带项目标签的 Issue 写入 `project_issues`（依赖主项目先有 issues_snapshot）。 |
| **脚本与模块** | `generate_daily_dashboard.py`、`sync_project_issues.py`；`project_manager/`（project_sync、dashboard_generator）。 |
| **状态** | ✅ 已在 feature_issue_and_kanban 实现；支持 --demo 与 --output-html；真实数据需 project_issues 或 issues_snapshot 中带项目标签。 |
